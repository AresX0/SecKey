📋 Overview & Features

SecKey is a comprehensive security and configuration platform for Azure/Entra ID, Intune, and endpoint hardening operations. It enables tenant-wide security deployments with detailed auditing, rollback capabilities, native analysis tools, and granular live-edit control over policy settings.

Core Capabilities

Full Environment Deployment

Deploy entire security environments including policies, groups, users, and infrastructure in a single operation.

Step-by-Step Control

Deploy individual components for granular control and testing.

Infrastructure Management

Manage authentication contexts, admin units, reusable settings, and platform scripts.

Deployment Auditing

Complete deployment history with detailed logging of all operations and outcomes.

Rollback Support

Undo deployments to safely revert changes to the tenant.

Export/Import Settings

Export existing tenant settings and import them back for consistency across environments.

🚀 Getting Started

Step 1: Authentication

Click "Sign In / Out" and authenticate with your tenant Global Administrator account.

⚠️ Security Note: You must have Global Administrator or equivalent role in Entra ID. Break-glass accounts are recommended.

Step 2: Dashboard

Review the Dashboard to verify your connection and see deployment status overview.

Step 3: Choose Deployment Method

Option A: Full Deployment (Recommended for new environments)
Go to "Deploy/Remove Secure Keyboard infra" and click "Apply Full SecKey Environment". This deploys all policies, groups, and infrastructure in the correct order.
Option B: Step-by-Step (Recommended for testing)
Go to "Compliance + Config" and select individual deployment steps from the dropdown. This lets you test each component.

Step 4: Monitor Deployment

Watch the Deployment History at the bottom of each tab. Each row shows:

  • Time - When the operation occurred
  • Scope - What component (e.g., "Policies", "Infrastructure")
  • Stage - Start, Complete, or Failed
  • Message - Details including Created/Updated/Skipped counts

🔄 Deployment Guide

Full Environment Deployment Order

The system automatically deploys components in this order to ensure dependencies are met:

  1. Users - Entra ID users (break-glass accounts)
  2. Groups - Entra ID groups for targeting
  3. Scope Tags - RBAC scope tags
  4. Named Locations - Geographic locations for CA policies
  5. Conditional Access - CA policies (requires groups and locations first)
  6. Device Enrollment Restrictions - Enrollment controls
  7. Device Configuration - OMA-URI and CSP settings
  8. Device Compliance - Compliance rules
  9. Device Settings Catalog - Modern Windows 11 settings
  10. Store App Assignments - Microsoft Store app distribution
  11. Intune Applications - Win32 and custom apps
  12. Autopilot - Device enrollment profiles
  13. Enrollment Status Page - Device setup experience
  14. Endpoint Security - Firewall, antivirus, account protection
  15. Proactive Remediation Scripts - Automated fixes
  16. Break-Glass Enforcement - Add users to break-glass groups

Partial/Step-by-Step Deployment

Go to Compliance + Config and select a specific step to deploy just that component. Useful for:

  • Testing changes before full deployment
  • Updating individual policies without redeploying everything
  • Troubleshooting specific components

Rollback/Undo

To undo deployments, click "Undo Full SecKey Environment". This:

  • Deletes all deployed policies and objects (by reverse order)
  • Removes users from break-glass groups
  • Skips non-destructive items like Store app assignments
⚠️ Caution: Undo is destructive. Ensure no critical policies are in use before running.

⏭️ Why Items Are Skipped During Deployment

The deployment engine skips items in these scenarios:

1. Item Already Exists (No Update Needed)

Reason: A policy with the same name already exists and is identical to the manifest definition.

Action: Skipped (no create, no update).

Example: "Windows 11 Security Baseline" policy already deployed last week and hasn't changed.

2. Store App Assignments (Non-Destructive)

Reason: Assigning apps to groups is additive. Removing them is not supported in this tool to prevent accidental unassignment.

Action: Skipped during undo operations (see note below).

ℹ️ To remove app assignments, use the Intune portal directly under Intune Apps → Manage → Assignment.

3. Authentication Context Policies In Use

Reason: Authentication strength policies referenced by Conditional Access policies cannot be deleted directly by the Graph API.

Action: Skipped during undo (silently, no error).

ℹ️ To delete, first remove references in Conditional Access policies, then manually delete in the portal.

4. File Format or Parsing Errors

Reason: JSON manifest file is malformed or missing required fields.

Action: Logged and skipped; deployment continues with other items.

Check: Review the deployment log for error messages.

5. Insufficient Permissions

Reason: Your authenticated account lacks permissions to create/delete this resource type.

Action: Error is logged; deployment continues. Check your Entra ID roles.

⚠️ Ensure your account has: Global Administrator or Policy Administrator + Compliance Administrator + Intune Administrator roles.

6. Duplicate Detection in Groups

Reason: A group member is already assigned to the same group.

Action: Skipped to avoid duplication errors.

Understanding the Deployment Message

Example: Created 27 items, updated 20, deleted 0, skipped 99.

TermMeaning
CreatedNew policies/objects created in the tenant.
UpdatedExisting objects with differences that were modified.
DeletedObjects removed from the tenant (undo only).
SkippedItems not acted upon (already exist, no changes, insufficient permissions, or by design).

High Skip Counts Are Normal

On second and subsequent runs, expect high skip counts because:

  • Policies are already created
  • Groups are already assigned
  • Users are already in place

This is safe and expected. You can run deployments multiple times without harm.

🔧 Troubleshooting

Deployment Fails: "Permission Denied"

Cause: Insufficient permissions.

Solution: Verify your account has Global Administrator or all required admin roles in Entra ID.

Deployment Fails: "Graph API Timeout"

Cause: Network issue or tenant load.

Solution: Check your network connection and try again in a few minutes.

High Skip Count

Cause: Policies already deployed.

Solution: This is normal on subsequent runs. No action needed.

Authentication Context Deletion Fails

Cause: Authentication strength policy is referenced by CA policies.

Solution: Remove references in Conditional Access policies first, then retry undo.

Cannot Delete Group

Cause: Group is still assigned to policies.

Solution: Remove group from all policy assignments first in the Intune portal.

App Upload Fails

Cause: .intunewin file is corrupted or oversized (>500MB).

Solution: Recreate the .intunewin using Microsoft Intune App Wrapping Tool.

❓ Frequently Asked Questions

Q: Is it safe to run the full deployment multiple times?

A: Yes. Items that already exist are skipped. Running multiple times is idempotent (safe).

Q: Can I undo just one component?

A: The GUI only supports full undo. To undo specific items, use the Intune portal directly.

Q: How do I modify a policy after deployment?

A: Edit the JSON manifest file in the JSON/ folder, then run the step-by-step deployment for that component. The system will update the policy.

Q: What if a deployment fails mid-way?

A: The system is resilient and continues deploying remaining items. Check the deployment history for the failure, fix the manifest, and rerun.

Q: Do I need to be connected to any VPN?

A: No, but break-glass accounts should only be used from controlled networks (Conditional Access may restrict access).

Q: Can I export my tenant's current settings?

A: Yes, use Import Exported Settings on the Infrastructure tab to import existing CA and Intune JSON exports.

Q: How often should I rerun deployments?

A: Only when you update manifests. After initial deployment, rerun only to apply policy changes.

Q: What does "Skipped for Import-StoreAppsEntraGroupAssignmentList" mean?

A: App assignments are "soft-deleted" - the system doesn't remove them during undo to prevent accidental unassignment. To remove, use the Intune portal.
